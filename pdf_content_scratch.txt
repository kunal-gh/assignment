Building an AI-Powered Resume Screening &
Candidate Ranking System
Executive Summary
This report outlines an end-to-end AI-driven pipeline to parse and rank job candidates by relevance to a job
description, using state-of-the-art NLP and ML techniques. The system extracts structured data (skills,
experience, education) from PDF/Word resumes, computes semantic similarity to the job description via
transformer embeddings, and optionally re-scores candidates with a Large Language Model (LLM). It then
displays an interactive Streamlit dashboard of the top candidates, complete with explainable insights. We
emphasize fairness and explainability (e.g. computing bias metrics and providing textual rationales), and
we employ robust engineering best practices (clean code, tests, CI/CD, Docker, documentation). Key
deliverables include a GitHub repo with clear architecture, a live demo link, and reproducible environments.
This project showcases both technical depth (embeddings, LLMs, vector search) and software rigor (project
structure, testing, Docker, pipeline), making your profile stand out as an AI/Data Science candidate
.
Technical Approaches for Screening & Ranking
Keyword vs Semantic Matching: Traditional ATS use rule-based or keyword matching (e.g. TF-IDF)
to score resumes by word overlap
. While fast, these brittle methods miss context (synonyms,
phrasing) and lack adaptability. Modern approaches embed text in vector space using models like
Sentence-BERT. For example, one study embeds resumes and job descriptions with SBERT and ranks
by cosine similarity
, achieving ~90% precision. We propose a hybrid scoring: combine exact skill/
keyword matching (or TF-IDF) with semantic similarity. A simple weighted formula might be: 
# Example scoring formula (weights can be tuned)
skill_score = matched_skill_count / total_required_skills
semantic_score = cosine_similarity(resume_embedding, jd_embedding)
final_score = 0.6 * skill_score + 0.4 * semantic_score
This leverages both human-defined criteria (skills, experience) and embeddings. Pre-trained embeddings
(e.g. all-MiniLM-L6-v2  or multi-qa-MiniLM-L6-cos-v1 ) are excellent at capturing resume-JD fit
. 
LLM Reranking / RAG: Beyond simple cosine scores, we can use an LLM (e.g. GPT-4 or a local
LLaMA) to refine or explain rankings. In a “multi-agent” framework
, one agent extracts data,
another scores/filters, and an LLM agent re-ranks or justifies matches. For example, the JobPilot
pipeline first retrieves top candidates via embedding+vector DB, then feeds them to Google Gemini
(LLM) to re-score and explain
. Likewise, a Retrieval-Augmented Generation (RAG) approach can
enrich scoring with company-specific criteria dynamically
. RAG makes the system context-
aware: it can load the relevant job or company context at query time without retraining.

Vector Database & Filters: To scale semantic search, we store resume embeddings in a vector DB
(Pinecone, Qdrant, Weaviate) for fast nearest-neighbor lookup
. Qdrant explicitly promotes a 
“Vector + Filter” architecture: perform semantic candidate retrieval while enforcing strict criteria
(location, certifications, etc.)
. For example, a hybrid pipeline first filters by mandatory fields
(e.g. work authorization) and then sorts the remainder by embedding similarity. This handles 
fuzziness in text (e.g. “ML” vs “Machine Learning”) while honoring hard constraints. Such a pipeline
can significantly improve hiring metrics: one company reported boosting fill rate from 20% to 48% by
vector-ranking candidates
. 
Figure: Example “vector+filter” matching pipeline (source: Qdrant HR Tech blog
). Resumes and job
descriptions are parsed and embedded, a semantic search retrieves candidates, strict filters enforce constraints,
and candidates are ranked by relevance.
Bias Mitigation & Explainability: Automated hiring must guard against bias. We will incorporate
fairness checks and transparent features. For instance, we can anonymize names/addresses or
calibrate scores across groups to satisfy an 80% impact ratio (EEOC’s four-fifths rule)
. Group
fairness metrics (e.g. demographic parity) and individual fairness (similar candidates get similar
scores) should be monitored
. We can use tools like Fairlearn (Microsoft) to compute fairness
metrics and apply mitigation
. For explainability, we will provide rationales for each score: either
via an LLM-generated summary (“Candidate A matches because of X, Y, Z”) or by using SHAP/LIME on
any learned components. The system will log feature contributions (skill overlaps, years experience,
semantic score) so recruiters can audit decisions.

Large Language Models: For explanation or complex matching, LLMs are invaluable. GPT-4 or

like LangChain or LlamaIndex can manage prompts and retrieval (we can feed the JD and resume
content to an LLM to ask “how well does this candidate fit?”).
Parsing and NLP: Extracting info from resumes is a solved (though messy) problem. Popular tools
include pdfplumber  or PyMuPDF for text extraction
, and python-docx  for .docx files
. For scanned images, Tesseract OCR can be used. We will likely rely on a combination of rule-
based and ML methods: e.g. regex or spaCy NER for emails, dates, and titles; dictionary matching for
skills; and fallback to an LLM if structure is highly variable. A recent GitHub parser (PyResume/
LeverParser) exemplifies extracting key fields with high accuracy
. We can also use spaCy (with
the en_core_web_sm  model or custom NER) to identify entities. For skills, a lookup against
industry lexicons (e.g. from O*NET) or an open-source skill extractor can speed things up.
Vector Databases & Search: Pinecone, Qdrant, and Weaviate are leading vector DBs for
embedding search
. They handle indexing millions of vectors and fast nearest-neighbor queries.
Pinecone is easy (hosted service), Qdrant is open-source/self-hosted, and Weaviate adds built-in
filters. For a quick MVP, Pinecone or Qdrant (free tier) can be used to store resume embeddings and
run similarity queries. Locally, FAISS or Annoy (Facebook) could serve as an alternative if external
services are not allowed. The Qdrant HR blog emphasizes that vector+filter queries can scale to “tens
of millions of vectors” with quantization for performance
.
Development Frameworks: The system itself will be written in Python. Key libraries include: 
sentence-transformers  (for embeddings), transformers  (for any custom model or
tokenization), torch  or tensorflow  for ML backend, and scikit-learn  for any classical
processing or metrics. For RAG workflows, LangChain or Haystack can orchestrate document
chunks and LLM calls. We will use Streamlit
 for the front-end dashboard, as it allows rapid
prototyping of data apps. Streamlit’s docs highlight its ease (“transform Python scripts into
interactive web apps in minutes”)
. For API integration, the OpenAI Python SDK or llama.cpp
could be employed. Task scheduling (e.g., embedding many resumes) can be parallelized via
multiprocessing or Celery if needed. Version control via Git and CI via GitHub Actions or similar will
ensure reliability. 
Additional Tools: We’ll containerize with Docker for reproducible environments (specifying exact
library versions). For project management: pytest  for unit tests, flake8 / black  for style,

libraries (Plotly, Altair) can enrich the dashboard with charts. Finally, leverage domain resources: job
and resume datasets (e.g. from Kaggle or sourcing sample resumes) for development and demo.

Engineering Best Practices (GitHub & Deployment)
Project Structure: Organize the repo clearly: e.g. src/  or package directories for core code, a 
data/  folder for sample resumes/job descriptions, a tests/  directory for unit tests, and a 
docs/  folder for design docs or diagrams. Include a concise README.md  with setup instructions,
usage examples, and an architecture overview (using text and a Mermaid diagram). Provide a 
requirements.txt  or environment.yml  so others can recreate the environment easily.
Documentation & Diagram: A well-documented architecture diagram helps recruiters and
developers. We will use Mermaid to embed a flowchart of the pipeline (see below). The README
should also explain the methodology: how resumes are parsed, what models are used, and how
scoring works. Inline comments and docstrings are a must. 
Data & Demo: Include a small set of sample resumes and a sample job description for
demonstration/testing. For privacy, use either fictitious or anonymized data. Providing a script or
notebook that runs end-to-end on these samples (from parsing to final ranking) will impress
recruiters by showing a working pipeline out of the box.
Testing & CI/CD: Write unit tests with pytest  to cover key functions (e.g. extract_skills() , 
compute_score() , etc.). Use GitHub Actions (or similar) to automatically run tests and lint on each
push. For example, we will create .github/workflows/ci.yml  to run pytest  and possibly
build a Docker image
. Automating checks (linting, type checks with mypy ) signals quality.
Continuous Deployment can be set up to auto-deploy the Streamlit app (e.g. to Heroku, Render, or
Streamlit Cloud) after tests pass
.
Reproducibility: Dockerize the app ( Dockerfile ) so that environment mismatches are
eliminated. The Dockerfile should pin versions (e.g., python:3.10 , streamlit==1.x , etc.). On
build, it can install all deps and test the app. We’ll also include a lock file (e.g. requirements.txt
or poetry.lock ) to freeze versions. For quick entry, consider a devcontainer.json  (VSCode
dev container) so others can replicate our dev setup.
Dashboard & UX: Ensure the Streamlit UI is clean and intuitive. Layouts with clear sections (upload,
input, results). Use progress indicators if processing is slow. Include error handling (e.g. “No text
found in PDF”). We will adopt Streamlit’s theming and markdown for a polished look. A video demo
link or GIF can be added to README to showcase the app interactively.

Evaluation Plan & Metrics
To demonstrate quality, we will evaluate the system on both accuracy and fairness. Key metrics include:
Rank-aware Accuracy: Use precision@K and recall@K to measure how many of the top-K ranked
candidates are truly qualified. For example, Precision@10 (top-10) indicates the fraction of those
shortlisted who were indeed suitable
. We will also track Mean Average Precision (MAP) and 
NDCG@K (Normalized Discounted Cumulative Gain) since they reward correct ordering
. ROC-
AUC is less useful for ranking but can be computed if framing as binary classification (fit vs. not fit)
.
Fairness Metrics: Compute group fairness measures. For any protected attribute (e.g. gender),
calculate the selection rate for each group and check the four-fifths rule (impact ratio ≥ 0.8)
.
Also check Equal Opportunity (true positive rates equal across groups)
. Tools like Microsoft’s
Fairlearn can compute these differences
. We will report any disparity and, if needed, adjust
scores (e.g. via thresholding or data augmentation) to mitigate bias.
Explainability/Calibration: Assess if scores are calibrated by comparing predicted “fit scores”
against actual outcomes (if labels exist). We can plot calibration curves or check expected vs.
observed hire rates for score buckets. We will also gather qualitative feedback: have recruiters
review a few candidate rankings and rate their relevance (blind evaluation). Inter-rater agreement
(Cohen’s kappa) with our model can be a benchmark.

Architecture Diagram
Below is a high-level Mermaid flowchart of the proposed system architecture. It shows how resumes and
the job description are processed, embedded, and ranked, with results shown in an interactive dashboard:
flowchart LR
    Resume[Resume PDFs/DOCX] --> TextExtract[Text Extraction (PyMuPDF, python-
docx)]
    JD[Job Description (Text)] --> TextExtract
    TextExtract --> Preproc[NLP Preprocessing (cleanup, skill/entity 
extraction)]
    Preproc --> Embed[Sentence-Transformer Embeddings]
    Embed --> DB[(Vector DB (Pinecone/Weaviate/Qdrant))]
    DB --> Search[Similarity Search (cosine)]
    Search --> Score[Compute Relevance Scores]
    Score --> TopN[Top-N Candidates]
    Score --> Explain[LLM-based Explanations / SHAP]
    TopN --> Dashboard[Streamlit Dashboard UI (ranking & insights)]
    Explain --> Dashboard
Each box would correspond to a module or service in code. For example, Text Extraction might use
PyMuPDF and spaCy, Embeddings uses a pre-trained SBERT model, Vector DB is a managed service, LLM
Explanations could be a prompt to GPT-4 or Llama, and Streamlit Dashboard ties it all together.
